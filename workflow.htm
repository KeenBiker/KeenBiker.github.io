JIRA workflow

Q: “How do tickets move thru different ticket states? What’s reqd at each stage?”

 

JIRA facilitates tracking of issue types. Core of-the-shelf types are Task and Subtask, whilst other categories that are available include Development (types: Epic, Story, Bug) and Service Desk (types: Incident, Change). User-defined, bespoke types can also be added. Issue types have associated project templates, that reflect typical underlying purpose, process and information changes. Available templates or schemes (collections of configured values), are based on JIRA products subscribed to: eg. if subscribed to Jira Service Desk, users can base projects on IT service desk and Customer service desk templates. Such templates provide configuration options for permissions, notifications, issue types etc., plus basic workflows.

 

JIRA workflows concern the stepwise progression of jobs or issues through an organisation’s processes. JIRA provides default workflows that can be copied & customised using workflow designer: a graphical workflow layout tool. This can create & edit statuses and transitions using drag-n-drop, specify screens, plus configure advanced transition options. The tool auto-validates workflows, highlighting statuses that have no incoming or outgoing transitions. JIRA workflows involve the following concepts …

 

  • Status – where an issue is within a particular workflow (eg. In Progress or Under Review). An issue can only be in one state at a time. Statuses are global, so changes in definition will manifest to all workflows that use a particular status.
  • Transition is a one-way link/move between two statuses that can be customised using key-value properties.
  • Resolution – why the Issue is no longer in flight (eg. because it’s completed).
  • Assignee refers to a person/actor/owner temporarily responsible for an issue. Assignment can be automatic via a post function, or surfaced via a transition screen, so the next issue owner can be chosen.
  • Conditions – control who can action a transition.
  • Validators – pre-transition checks, eg. only allow transitions to occur when specific criteria are satisfied.
  • Post Functions – make additional changes to issues following transitions (eg. remove a resolution if issue is reopened, or setting assignee: Unassigned)
  • Triggers – auto-transition on an event. Eg. auto-move from In Progress to Under Review when code is submitted for review.
  • Workflow properties – set properties for transitions. Eg. only display resolutions that are relevant to a specific Issue type.
  • Workflow schemes – associate workflows and issue types.

 

Some top tips for designing JIRA workflows

1.    Gather all of your stakeholders. Workflow is described as scaling culture, about people, so to build a process around a set of people, first identify all involved stakeholders. Spend time talking with each one to understand their priorities, then draft a whiteboard workflow & collectively walk through use-cases, refining via feedback.

2.    KISS (Keep it simple). Each status adds transitions, conditions & complexity: Whenever adding a new status, make sure you have no other option, like returning an issue to a previous state.

3.    Treat workflow development like iterative software development: discuss, design, validate against use-cases, deploy, track deviations, review, refine . . .

 

In proposing an idealised JIRA-based workflow, it’s worth noting some criticism of JIRA: that allowing a tool to define our process is the real antipattern. . . Jira on its own doesn’t have the capabilities for proper project management. . . the process of breaking down work into small tasks and assigning them to team members makes their focus shift too much to the ‘micro’ side and forget about the ‘macro’. Without the context . . . everyone cares only about their own issues, so the teamwork becomes just running the tickets through the board . . . reports consist mostly of velocity and ticket burndown charts, which often don’t show the actual amount of value brought to the customer.

 

Elsewhere: JIRA tickets come with built-in assumptions . . . that the feature / behaviour described is discrete. That it is ultimately binary, i.e. either complete or not. That it can be individually estimated. That it can be worked on in relative isolation. That its connections to other tickets can be modelled very simply, using JIRA’s childishly simple notion of linked tickets . . . (devs) are implicitly pushed to architect software in a way that maps to the JIRA tickets, even if this isn’t technically optimal.

 

Or here: JIRA implicitly teaches everyone to ignore the larger vision while focusing on details. There is no whole. At best there is an Epic, but the whole point of an Epic is to be decomposed into smaller pieces to be worked on independently. JIRA encourages the disintegration of the macro vision.

 

Factoring such misgivings into a proposed workflow for JIRA-based code development, I suggest emphasis be given to this macro vision, to group understanding (particularly for new joiners) and to providing big picture context for new tasks, as well as specific objectives or actual test criteria against which code for review can be measured. . .
