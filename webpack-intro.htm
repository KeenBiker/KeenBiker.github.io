webpack: an overview, with attention to upcoming enhancements

NB: webpack is a product name intentionally lacking capitalisation.

 

What is webpack?

webpack is a module bundler like Browserify or Brunch. It is not a task runner like Make, Grunt, or Gulp that automate common development tasks such as linting, building, or testing projects. webpack specifically addresses the needs of front-end developers to optimally deploy their code & resources from /src folders to /dist or /build directories. Using wp, custom loaders can target multiple file types and its dependency graphs enable code-splitting (partitioning a codebase into multiple chunks, loaded asynchronously & on-demand at runtime), deduplicating frequently used code, eliminating unused code (tree-shaking) and minifying output bundles. Webpack thus maximises control of what’s loaded initially, what’s loaded at runtime, speedy run-time execution, separate prod & dev builds, even how code is “chunked” and hashed for browser caches. It can be used from the command line, or configured using webpack.config.js.

 

webpack architecture

webpack works with modules: discrete & coherent chunks of functionality, providing solid abstractions and encapsulation boundaries, that simplify debug & testing. With JS and Node, use of modules is characterised by import and require() statements. webpack is built around an event-driven pipeline, with a Compiler instance as its main engine & central dispatcher, configured using options passed via the CLI or Node API. Loaders access file resources. Everything else is done by plugins. These are task-orented code modules, each with an apply() method that the compiler calls via a hook(), plus various Tapable and interceptor API methods. Hence, plugins attach themselves to events, which correlate to the different phases of the build.

 

A webpack use-case normally initiates via a webpack.config.js file that specifies user-defined entry points. Paths/file access is resolved and inputs handled by regex-based rules that match dedicated loaders to particular file types. Dedicated factory components take new (previously unseen) loaded modules & instance parsers that convert source code to Abstract Syntax Trees (AST). webpack traverses each AST, identifying all require & import statements to generate a dependency graph. These graphs are targetted at builds. Production builds require fast bundle load times/compressed bundles, hence webpack provides a plugin that applies UglifyJS. Development mode requires speedy compilation & output. It too uses Uglify, but adds deeper source-mapping, partial rebuilds and a localhost server (webpack-dev-server) with live reloading & module replacement. Separate prod/dev configurations are recommended, plus a third for common elements (add using webpack-merge).

 

Runtime caching

Once the contents of /dist have been deployed to a server, clients will attempt to display the site, a relatively slow step that browsers optimise by caching content, which wp can assist. Since third-party libraries such as lodash or React are less likely to change than local source code, webpack enables code-splitting to separate vendor chunks. This allows up-to-date clients to request much less from server. webpack v4 enhanced this process by adding cacheGroups to SplitChunksPlugin, hence support for prefetching (of resources probably needed for a future navigation) and preloading (of resources possibly needed during current navigation).

 

webpack Vsn 5 Enhancements

Module federation that allows dynamic, bi-directional, run-time sharing of JS code & dependencies between independent bundles, builds and whole applications, on both client and server.

Persistent caching promises to improve build start up performance, plus allow caching on disk (or other persistent mediums). Caches have previously been auto-assigned temporary hashes, meaning webpack resolved filenames via a map of module IDs to hashes. By default, webpack included this digest in the boilerplate code added to the top of every bundle. This is reported problematic, as the digest had to be updated every time a module was added or removed and when the digest changed, whole bundles had to be rebuilt. New bundles broke cache references, forcing fresh downloads onto customers. New in webpack 5, options include: use of memory or specified filesystem location; human readable (non-numeric) cache naming, storage & versioning; choice of hash algorithm (prod only). NB: Persistent caching is more expensive due to serialisation and disk allocation, so should be run only when compiler is idle or otherwise closing.

Module Size values: Vsn 5 modules express size differently: richer than just a single number. The SplitChunksPlugin now knows how to handle these different values to impose minSize and maxSize bundle limits.

 

Bundle Analysis

Whilst splitting code, it may be useful to visualise results. There’s an official analyze tool, plus other community-supported options…

webpack-chart: Interactive pie chart for webpack stats.

webpack-visualizer: Visualize and analyze your bundles to see which modules are taking up space and which might be duplicates. $ npm install webpack-visualizer-plugin

webpack-stats-plugin: ingest the webpack stats object, process / transform the object and write out to a file for further consumption. The most common use case is building a hashed bundle and wanting to programmatically refer to the correct bundle path in your Node.js server.

webpack-bundle-analyzer: A plugin and CLI utility that exposes bundle content as a convenient interactive zoomable treemap. Also works with minified bundles. The visualisation can be filtered (on chunks) and is intended to assist optimisation.   

webpack bundle optimize helper: On-line tool that accepts a user’s wp bundle stats file as an upload, then provides actionable suggestions to reduce bundle size.

bundle-stats: Generate bundle reports (size, assets, modules) to compare results between different builds.

 

Improving build speeds

Official tips are provided here: use include fields, so loaders are applied to the minimum number of modules necessary; offload expensive loaders to a worker pool; minimise resolve requirements; keep chunks small; removing the ProgressPlugin from the configuration; compile in memory (webpack-dev-server)…

 

Proactively improving build speeds requires benchmarks. Node ships with an inspector that can profile builds, plus Google explains how to profile in Chrome in great detail.

 

Workloads can be reduced, eg. the Slack team report how setting UglifyJsPlugin compression options to false made minification 300% faster, yet left bundle sizes barely changed. Another expensive step is AST building. If a file (or set of files) will never use import, require, or define statements, webpack can exclude them from this process. Skipping large libraries in particular can really boost performance. See the noParse option for more detail.

 

Workloads can also be distributed: parallel-webpack will perform whole webpack builds in parallel. happypack will execute loaders in parallel as will thread-loader, an equivalent written and maintained by the core webpack team. Worker Farm on npm distributes task processing to child processes using a simple API with custom concurrency options. This looks useful for bundle builds across multiple cores, as seen here.
